Introduction API REST

Le développement Backend repose sur les API (Application Programming Interface), qui permettent de faire le lien entre vos données sécurisées et vos utilisateurs web.

L'analogie du restaurant

Le Client (navigateur) est le client du restaurant. Le Serveur est le cuisinier. L'API est le serveur en salle : il prend la commande, la confie à la cuisine et ramène l'assiette au client.

Les requêtes REST standards

Toute base de données peut être manipulée via 4 opérations CRUD (Create, Read, Update, Delete) reliées aux commandes réseau HTTP.

GET

Demande au serveur de *lire* simplement les données pour vous les renvoyer.

POST

Envoie de toutes nouvelles données au serveur pour *créer* un document.

PUT / PATCH

Indique au serveur de modifier ou *mettre à jour* une donnée existante.

DELETE

Demande au serveur la suppression brute des ressources ciblées.

Le format JSON

Les serveurs et les interfaces discutent généralement dans un format extrêmement simple, composé de clés et de valeurs : le format de fichier JSON (JavaScript Object Notation).

// Réponse typique d'une API à une requête GET /users/1
{
  "id": 1,
  "nom": "Dupont",
  "prenom": "Alice",
  "role": "Admin",
  "verifie": true
}

Les Codes d'État (HTTP Status Codes)

À chaque fin de cycle, le serveur renvoie un code chiffré permettant à la machine client de savoir si tout s'est bien passé.

  • 2xx (Succès) : 200 OK signifie que l'opération a réussi, 201 Created qu'une ressource (compte, objet) a bien été générée.
  • 4xx (Erreur client) : 400 Bad Request quand la donnée envoyée est corrompue, 404 Not Found quand l'adresse API n'existe pas ou 401 Unauthorized si on n'a pas les droits requis.
  • 5xx (Erreur serveur) : 500 Internal Server Error prouve de manière indéniable que la faute incombe au développeur backend : l'application qui gère la base a complètement planté.
